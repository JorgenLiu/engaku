## Background

The v1.1.0 release shipped with several bugs, to be addressed in v1.1.1: (1) the template `engaku.json`
still maps `"dev"` instead of `"coder"`, causing `engaku apply` to skip the
coder agent and lose its model field; (2) `@dev` references remain in the
reviewer template; (3) output formatting alignment differs between `init`,
`update`, and `apply`. Additionally, `engaku inject` only finds the first
in-progress task, silently ignoring others — a problem when multiple tasks
are active simultaneously.

## Design

### Bug fixes (1–3)

1. **Template `engaku.json` key rename** — Change `"dev"` → `"coder"` in
   `src/engaku/templates/ai/engaku.json`. Also fix this repo's own
   `.ai/engaku.json`. This resolves both the "dev remnant" bug and the
   "coder model disappears" bug (apply couldn't match the key to the file).

2. **Reviewer `@dev` → `@coder`** — Two occurrences in the reviewer
   template body (lines 23 and 75). Update both the template version
   (`src/engaku/templates/agents/reviewer.agent.md`) and the live version
   (`.github/agents/reviewer.agent.md`).

3. **Consistent output alignment** — Standardize all three commands to
   10-char left-aligned tags. Current state:
   - `init` uses 9-char: `[skip]···`, `[create]·`
   - `update` uses 10-char: `[update]··`, `[create]··` but calls init's
     `_copy_template` (9-char) for the lessons stub
   - `apply` uses 2-space indent + different tags: `··[updated]·`, `··[skip]····`

   Target format (10-char, no indent):
   `[skip]    ` `[create]  ` `[update]  ` `[updated] `

   In `cmd_apply.py`, also remove the 2-space leading indent to match the
   other commands.

### Multi-task injection (enhancement)

Change `_find_active_task` (singular) → `_find_active_tasks` (plural).
Returns a list of `(title, unchecked, body, filename, state)` tuples for
**all** `.ai/tasks/*.md` with `status: in-progress`.

Classification per task:
- **needs-work** — has at least one `- [ ]` line
- **needs-review** — all checkboxes are `[x]` (or no checkboxes exist)
  but status is still `in-progress`

Injection format (replaces current `<active-task>` singular):

```xml
<active-tasks>
<task file="2026-04-20-feature.md" state="needs-work">
## Feature Title
- [ ] Step 3: Implement validator
- [ ] Step 4: Write tests
</task>
<task file="2026-04-18-refactor.md" state="needs-review">
## Refactor Title
All tasks completed. Awaiting reviewer verification.
</task>
</active-tasks>
```

For `PreCompact`, each task includes Background/Design/File Map + all
checkboxes (same depth as current, but per-task).

When no tasks are in-progress, no `<active-tasks>` block is emitted (same
as current behavior with `<active-task>`).

Agent-agnostic: all agents receive the same injection. `SessionStart` and
`PreCompact` hooks don't receive agent identity (confirmed via VS Code
docs); agents self-select what matters based on their system prompts.

See `.ai/docs/multi-task-injection.md` for detailed rationale.
