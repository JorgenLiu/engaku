## Background

v1.0.0 is the stable milestone release of engaku. Per decision 003, this is the
final release with `requires-python = ">=3.8"`. After v1.0, the project bumps to
Python 3.11 and adds MCP integration (decision 004).

Two items require code changes before tagging v1.0:

1. **Scanner granularity**: The scanner agent currently instructs agents to
   "cluster files into 3–6 logical groups." For large codebases (50+ source
   files) this produces meaninglessly broad groups with overlapping conventions.
   The hard limit must be replaced with a scale-aware heuristic.

2. **Version bump and CHANGELOG**: `pyproject.toml`, `__init__.py`, and
   `CHANGELOG.md` must reflect the 1.0.0 release. The README should note that
   v1.0.x is the pinnable baseline for Python 3.8 environments.

## Design

### Scanner granularity fix

The root problem: "3–6 groups" was chosen to prevent one-group-per-file
over-engineering, but the floor of 3 and ceiling of 6 are arbitrary and break
down at scale.

**New heuristic**: Remove the numeric bounds. Instead, instruct the scanner to:
- Propose as many groups as the codebase's distinct responsibility boundaries
  naturally suggest
- Aim for groups where a single `.instructions.md` body (< 40 lines) can
  meaningfully describe the conventions for all files in the group
- Never create a group for a single file unless that file is architecturally
  isolated (e.g. a standalone CLI entry point)
- For small repos (< 15 source files) a minimum of 2 groups is sufficient; for
  larger repos there is no cap

This change must be applied to **both** the live agent
(`.github/agents/scanner.agent.md`) and the template
(`src/engaku/templates/agents/scanner.agent.md`).
